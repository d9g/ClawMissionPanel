## 核心原则

### 1. 务实优先于流程 🎯

**教训**: 多 Agent 协作遇到阻塞时，持续催促不如直接代劳

**准则**:
```
1. 正常情况 → 多 Agent 协作（并行高效）
2. 阻塞/延期 > 12 小时 → 评估是否代劳
3. 用户明确要求 → 立即接手完成
```

**应用**:
- 不要为了流程而流程
- 当流程成为阻碍时，果断简化
- 结果导向，不是过程导向

---

### 2. 主动分析，不要被动等待 ⚡

**教训**: 用户指出问题才行动 → 太晚了

**准则**:
```
看到问题 → 主动分析 → 提出建议 → 等用户确认 → 执行
```

**应用**:
- 识别功能重叠时，立即提出整合建议
- 发现任务延期时，主动分析根本原因
- 看到状态不一致时，立即同步更新

---

### 3. 状态透明化 📊

**教训**: 任务文件、数据库、页面显示三者不同步

**准则**:
```
任何状态变更 → 同时更新所有副本 → 验证一致性
```

**应用**:
- 任务完成 → 更新任务文件 → 同步数据库 → 刷新页面
- 进度变化 → 立即记录 → 自动同步 → 实时显示
- 发现不一致 → 立即修复 → 记录原因

## 架构设计

### 4. 定时任务不能依赖 inbox 通知 ⏰

**教训**: 监控任务依赖 inbox 通知，inbox 为空后不再唤醒

**准则**:
```
❌ 错误：创建任务 → 等待 inbox 通知 → inbox 为空 → Agent 休眠
✅ 正确：创建任务 → 配置定时 → 到点自动唤醒 → 执行监控
```

**实施要点**:
1. 监控频率配置到唤醒服务 (gateway-agent-wakeup.js)
2. 每分钟检查，到点自动唤醒
3. 监控完成自动更新进度
4. 过时 30 分钟自动告警

---

### 5. 多 Agent 系统需要调度器 🎯

**教训**: 纯被动 Agent 无法协作，任务分配效率低

**准则**:
```
用户 → 调度器 (主动) → 任务分发 → Agent (被动响应)
                        ↓
                   状态收集 → 进度追踪 → 自动告警
```

**应用**:
- 基于多 Agent 调度最佳实践
- 避免重复造轮子
- 任务分配自动化

---

### 6. API 路径设计要一致 🔗

**教训**: Nginx rewrite 规则错误导致 404，前端 API 路径混乱

**准则**:
```
1. 后端 API 统一前缀：/api/...
2. Nginx 代理明确 rewrite 规则
3. 前端调用路径与后端一致
4. 版本号强制刷新缓存
```

**标准格式**:
```
后端：/api/tasks/stats
Nginx: /task-board-api/(.*) → /api/$1
前端：/task-board-api/tasks/stats
```

## 任务管理

### 7. 任务状态及时更新 📝

**教训**: 任务完成后未更新状态，导致任务板不准确

**准则**:
```
任务完成 → 立即更新状态 → 同步数据库 → 通知相关人员
```

**执行标准**:
- 完成后 5 分钟内更新状态
- 进度变化实时更新
- 发现状态错误立即修复

---

### 8. 监控任务按检查次数增长进度 📈

**教训**: 监控任务进度难以量化

**准则**:
```
进度 = (已完成检查次数 / 总检查次数) × 100%
```

**示例**:
| 时间 | 类型 | 进度 |
|------|------|------|
| 11:20 | 首次测试 | 30% |
| 15:30 | 第 1 次检查 | 75% |
| 16:30 | 第 2 次检查 | 80% |
| 19:30 | 第 3 次检查 | 85% |
| 23:30 | 最终报告 | 100% |

---

### 9. 依赖关系检测要准确 🔗

**教训**: DEP-A 已完成但 DEP-B/C 仍显示 PENDING

**准则**:
```
依赖检测逻辑：
1. 查询依赖任务状态
2. 检查是否为 COMPLETED
3. 更新依赖任务状态为 AVAILABLE
4. 通知执行者可以开始
```

**修复优先级**: P0（阻塞其他任务）

## 代码质量

### 10. SQL 查询要小心 ⚠️

**教训**: WHERE 排除条件误删重要数据

**准则**:
```
❌ 错误：WHERE status != 'COMPLETED'  -- 可能误删其他状态
✅ 正确：WHERE status IN ('PENDING', 'IN_PROGRESS')
```

**应用**:
- 优先使用 IN 而不是 !=
- 测试时覆盖边界情况
- 审查时重点关注 WHERE 条件

---

### 11. 功能开关优先于删除代码 🔧

**教训**: 需求变化需要移除功能时，删除代码后需求又回来了

**准则**:
```
需求变化 → 添加功能开关 → 关闭功能 → 保留代码
```

**好处**:
- 支持快速切换
- 支持 A/B 测试
- 需求反复时快速恢复

---

### 12. 文档与代码同步 📚

**教训**: 文档滞后于代码，导致信息不准确

**准则**:
```
代码变更 → 同步更新文档 → 验证链接有效 → 提交文档
```

**文档清单**:
- API 变更 → 更新 API.md
- 配置变更 → 更新 DEPLOYMENT.md
- 功能变更 → 更新 USER-GUIDE.md

## 监控运维

### 13. 服务挂了立即知道 🚨

**教训**: 用户反馈"服务挂了都不知道"

**准则**:
```
检查频率：每 1 分钟
响应时间：<2 分钟
自动恢复：尝试重启
失败告警：立即通知用户
```

**监控脚本**:
```bash
*/1 * * * * <workspace>/scripts/service-health-check.sh
```

---

### 14. 唤醒≠执行 ✅

**教训**: 唤醒服务显示"唤醒成功"但 Agent 未执行任务

**准则**:
```
❌ 错误：发送消息 → 显示"唤醒成功" → 结束
✅ 正确：发送消息 → 等待确认 → 验证执行 → 完成
```

**实施要点**:
- Agent 收到消息后必须确认
- Agent 开始执行后必须报告
- Agent 完成后必须更新状态
- 超时未确认自动重试
- 多次失败自动告警

---

### 15. 缓存更新要强制刷新 🔄

**教训**: 浏览器缓存旧版本 JS 导致功能异常

**准则**:
```
文件变更 → 更新版本号 (?v=YYYYMMDDHHmm) → 强制浏览器重新加载
```

**应用**:
- HTML 引用 JS/CSS 时带版本号
- 文件更新时同步更新版本号
- 提示用户强制刷新 (Ctrl+F5)

## 协作流程

### 16. 评审流程本身不需要被评审 ⚖️

**教训**: 设计评审流程的任务不需要走评审流程

**准则**:
```
设计类任务 → 输出完整文档 → 创建开发任务 → 标记完成
开发类任务 → 实现功能 → 测试验证 → 评审 → 完成
```

**应用**:
- 避免形式化的评审
- 评审聚焦于关键决策
- 小改动可以免评审

---

### 17. 任务分配要及时 ⏱️

**教训**: 等待任务未及时发现和分配

**准则**:
```
发现等待任务 → 立即分配 → 创建通知 → 更新状态
```

**执行标准**:
- 每 2 小时检查一次等待任务
- 分配后 5 分钟内确认执行者收到
- 超过 4 小时未开始 → 催促

---

### 18. 根本原因分析要文档化 📋

**教训**: 同样的问题反复出现

**准则**:
```
问题发生 → 根因分析 → 制定预防措施 → 文档化 → 分享团队
```

**文档模板**:
- 问题描述
- 根本原因 (5 Why 分析)
- 解决方案
- 预防措施
- 经验教训

## 避坑指南

### ️ 常见陷阱

| 陷阱 | 症状 | 解决方案 |
|------|------|---------|
| **数据库锁定** | 无法更新任务状态 | 先更新文件，等服务重启后同步 |
| **API 404** | 页面显示正常但数据为 0 | 检查 Nginx rewrite 规则和前端 API 路径 |
| **Agent 假死** | 唤醒成功但无响应 | 建立健康检查和自动告警机制 |
| **缓存问题** | 修复后页面仍显示旧数据 | 更新版本号，强制刷新 |
| **依赖检测 bug** | 前置任务完成但后续任务仍阻塞 | 修复依赖检测逻辑，手动触发同步 |

## 总结

### 🎯 核心思想

> **务实、主动、透明、自动化**

- **务实**: 结果导向，流程服务于目标
- **主动**: 提前分析，不要等问题暴露
- **透明**: 状态实时同步，信息准确可靠
- **自动化**: 能自动的不要手动，能定时的不要等待
